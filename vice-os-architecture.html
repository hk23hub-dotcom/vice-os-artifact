01 — MENTAL MODEL
The 3-layer model.
Three layers. Everything above depends on everything below. Never rebuild the bottom layer per vertical.
SYSTEM LAYERS
┌──────────────────────────────────────────────────────┐
│                  PRESENTATION LAYER                  │
│   Web App · Mobile App · Universe · Public Pages     │
├──────────────────────────────────────────────────────┤

│   ViceGolfer · Rugby Vice · Crypto Vice · HK23       │
│   Each vertical owns its own logic, screens, data    │
│   but CONSUMES shared modules from the Core Layer    │

1
One identity traverses everything
A user logs in once. Their profile, stats, memberships, collectibles and history exist across all verticals automatically.
2
Verticals are modules, not separate apps
ViceGolfer is a module inside VICE OS. It can be deployed standalone but always runs on Core. No duplicate auth, no duplicate billing.
3
Data flows upward, not sideways
A golf session generates data → Content Engine processes it → rankings update → vault archives it. Verticals never talk to each other directly.
02 — FOLDER STRUCTURE
Monorepo.
Modular by design.
A Turborepo monorepo. Shared packages are consumed by all apps. Adding a new vertical = adding a folder inside apps/web/app/(verticals)/ and wiring it to the shared packages. Zero structural changes required.

04 — DATABASE SCHEMA
Supabase / PostgreSQL.
One source of truth.
13 core tables. Each vertical adds its own session/profile tables. Shared tables never change when a new vertical is added.
CORE TABLES
-- ─── IDENTITY ───────────────────────────────────────
users (
  id uuid PRIMARY KEY,
  email text UNIQUE NOT NULL,
  username text UNIQUE,
  full_name text,
  avatar_url text,
  bio text,
  membership_tier text DEFAULT 'starter',   -- starter|pro|elite|founder
  sport_access text[] DEFAULT '{}',          -- ['golf','rugby','crypto']
  vice_level int DEFAULT 0,                  -- cross-vertical XP
  collectibles_count int DEFAULT 0,
  is_founder boolean DEFAULT false,
  created_at timestamptz DEFAULT now()
)

-- ─── PERFORMANCE DNA ─────────────────────────────────
dna_profiles (
  id uuid PRIMARY KEY,
  user_id uuid REFERENCES users,

  updated_at timestamptz DEFAULT now()

16 — ARCHITECTURE RULES
The laws
of VICE OS.
These rules exist so that every decision — now and in 3 years — is consistent. Break one and the system starts to feel generic.
I
One identity. Never duplicated.
No per-vertical user tables. No second auth system. No per-vertical billing. The Core owns all of this. Verticals consume it.
NON-NEGOTIABLE
II
Never hard-block. Always upsell.
A user without access sees a blurred/locked state with a clear path to unlock. "Access denied" destroys desire. "You're 1 step away" creates it.
UX LAW
III
Session data is the raw material for everything.
Every session recorded feeds DNA, content engine, vault, rankings, and AI simultaneously. If a feature doesn't connect to session data, question whether it belongs.
DATA LAW
IV
Ship, then polish. Never the reverse.
A working kick tracker with basic UI beats a perfect kick tracker that ships in 3 months. VICE OS is a living system. Polish is a feature, not a prerequisite.
EXEC LAW
V
The design language is the brand.
If a screen could exist in another app, it doesn't belong in VICE OS. Every screen must feel like it came from the same cinematic, performance OS. No exceptions.
DESIGN LAW
VI
AI is context, not a chatbot.
